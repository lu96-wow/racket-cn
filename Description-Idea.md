# Description-Idea — racket-cn 设计思路

## 一、`no-translate`：翻译器的转义机制

### 1.1 问题的产生

翻译器做的是全量标识符替换：遇到 `定义` 就换成 `define`，遇到 `hash` 就换成 `哈希`。这在一个前提下是正确的——代码中出现的每个标识符都属于"将被求值的命名空间"。

但 `quote` 打破了这一点。同一个语法形式在不同上下文里有完全不同的身份：

```racket
;; 场景 A：'hash 是哈希表的键（数据）
(hash 'hash 'a 'bash 'b)

;; 场景 B：'(哈希 'a 'b) 会被 eval 求值（代码）
(eval '(哈希 'a 'b))
```

在场景 A 里 `'hash` 应该保留为符号 `hash`；在场景 B 里 `'哈希` 需要翻译为 `'hash` 才能在标准 Racket 里被 `eval` 正确执行。

翻译器只拿到一棵裸语法树——它看到两个 `(quote ...)` 形式，但不知道一个流向 `hash` 构造器，另一个流向 `eval`。这是静态分析的固有限制（等价于停机问题）。

### 1.2 讨论过程

最初考虑过"保守策略"——quote 内部一律不翻译。理由是大多数 `quote` 用法确实是数据（hash key、符号常量、配置表），eval 场景相对少见。

但很快意识到 `#lang racket-cn` 本身的语义反驳了这一点：

```racket
#lang racket-cn
(eval '(哈希 'a 'b))     ;; ✅ 能跑——'哈希 就是符号 哈希，绑定到了 哈希 函数
```

在 CN 环境中，`'哈希` 是符号 `哈希`，求值时找到的是中文命名空间。翻译成标准 Racket 后，如果 `'哈希` 被保留为 `'哈希`，eval 会在标准 Racket 命名空间里找 `哈希` 这个符号——找不到，失败。因此翻译器**必须**把 `'哈希` 翻译为 `'hash`。激进的全量翻译才是语义正确的。

既然默认行为是正确的，剩下的问题是：用户什么时候需要"不要翻译"？

答案是：当用户把一个符号纯粹当作数据键使用，而这个键碰巧和某个 Racket 函数同名的时候。`(hash 'hash 'a)` 中，`'hash` 是键不是函数调用，翻译成 `'哈希` 会让数据变味。

### 1.3 设计方案

提供一个编译期标注 `no-translate`：

```racket
(hash (no-translate 'hash) 'a
      (no-translate 'bash) 'b)
```

翻译器遇到 `(no-translate expr)` 时：
1. 剥掉 `no-translate` 包装
2. 返回 `expr` 原样，不对其做任何标识符替换

翻译后：
```racket
(哈希 'hash 'a
      'bash 'b)
```

`no-translate` 是显式的、局部的转义——不影响其他代码的翻译，也不会出现在输出里。

### 1.4 实现

在翻译器的递归语法树遍历中，`syntax-case` 的 pair 匹配增加一个 literal 模式：

```racket
(syntax-case stx (no-translate)
  ;; (no-translate expr) → 直接返回 expr，不递归翻译
  [(no-translate expr)
   #'expr]
  [(a . d)
   ;; ... 正常递归翻译 ...])
```

`(no-translate expr)` 比通用 `(a . d)` 优先匹配。`no-translate` 本身不在标识符映射表中，因此不会被翻译，在两种方向下都能正确识别。


## 二、`kw.rkt`：关键字参数翻译宏

### 2.1 为什么不能只用 `rename-out`

`rename-out` 是 racket-cn 翻译普通标识符的核心机制：

```racket
(provide (rename-out [define 定义]
                     [lambda 函数]))
```

这能把 `定义` 重定向到 `define`，把 `函数` 重定向到 `lambda`。但它**只对标识符有效**。

Racket 的关键字参数（`#:exists`、`#:mode` 等）是独立于标识符的句法类别。`rename-out` 无法重命名关键字，因为关键字在 Racket 的展开器里是作为"关键字实参"被解析的，不参与标识符绑定。所以 `(rename-out [#:exists #:如果存在])` 是不可能的。

这意味着：不能直接让用户写 `(打开输出文件 "a.txt" #:如果存在 '截断)` 然后通过简单的重命名映射到 `(open-output-file "a.txt" #:exists 'truncate)`。

### 2.2 设计思路

`定义-关键字函数` 做的事情是：**定义一个宏，在编译期把带中文关键字参数的调用改写为带英文关键字参数的调用**。

使用者写一行：

```racket
(定义-关键字函数 打开输出文件 open-output-file
  (#:如果存在 #:exists (截断 truncate) (替换 replace))
  (#:模式   #:mode   (文本 text)))
```

这会展开成一个 `define-syntax`，名字叫 `打开输出文件`。当代码中出现：

```racket
(打开输出文件 "a.txt" #:如果存在 '截断 #:模式 '文本)
```

这个宏会遍历参数列表，将：
- 关键字 `#:如果存在` → `#:exists`
- 关键字 `#:模式` → `#:mode`
- 引述值 `'截断` → `'truncate`（仅当它跟在 `#:如果存在` 后面时）
- 引述值 `'文本` → `'text`（仅当它跟在 `#:模式` 后面时）

然后把参数打包，生成对英文函数的调用：

```racket
(open-output-file "a.txt" #:exists 'truncate #:mode 'text)
```

### 2.3 宏展开生成的结构

`定义-关键字函数` 本质上是一个**宏生成器**——它读声明式的规范，吐出 `define-syntax` 代码。展开后的宏包含三个关键部分：

**① kw-map — 关键字映射表**

`((#:如果存在 . #:exists) (#:模式 . #:mode))`

遍历参数时，遇到关键字就查这张表，找到就替换。

**② val-map — 关键字值映射表**

`((#:exists ((截断 . truncate) (替换 . replace))) (#:mode ((文本 . text))))`

每个被翻译的关键字关联一个取值映射。当遍历器看到一个带引号的符号且前一个元素是已知关键字时，用这张表查找翻译。

**③ 遍历状态机 — `k` 上下文**

```racket
(let loop ([rest lst] [k #f] [acc '()])
  (let ([d (syntax->datum (car rest))])
    (cond
      [(keyword? d)
       ;; 查 kw-map，记录当前关键字上下文 k
       ...]
      [(and k (pair? d) (eq? (car d) 'quote) ...)
       ;; 有 k 上下文时，'截断 → 'truncate（用 val-map 查）
       ...]
      [else
       ;; 普通参数原样通过
       ...])))
```

`k` 是关键设计：当前面出现 `#:如果存在` 时，`k` 被设为 `#:exists`；下一个元素如果是 `'截断`，用 `k` 去 `val-map` 找 `#:exists` 对应的子表，再查 `截断 → truncate`。遍历完这个参数后，`k` 重置为 `#f`，避免影响后续参数。

这确保了 `(打开输出文件 "a" #:如果存在 '截断 #:模式 '文本)` 中 `'截断` 只在 `#:如果存在` 的上下文中被翻译，`'文本` 只在 `#:模式` 的上下文中被翻译——两者不会互相干扰。

### 2.4 为什么不用更简单的方案

另一个可能的方案是：让用户在调用时直接用英文关键字。例如：

```racket
(打开输出文件 "a.txt" #:exists 'truncate)  ;; 关键字已经是英文
```

这在技术上可行——`创建-关键字函数` 生成的宏对英文关键字是透传的。但这样丧失了"中文 Racket"的初衷——如果关键字参数还是要写英文，语言体验就不完整。`kw.rkt` 的价值是让用户**全程用中文**调用函数，包括关键字参数。

### 2.5 一个理论上的边界冲突

`kw.rkt` 只做顶层参数遍历，不递归进入嵌套表达式。但在**顶层平面**上存在一个判定：

`(car d)` 等于 `quote` 且紧随已知关键字，就认为该参数是关键字值并翻译。这在所有正常使用场景下都是正确的——`#:如果存在 '截断` 中 `'截断` 确实是关键字实参。

如果出现以下极端情况，翻译就会出错：

```racket
;; 某个函数接受 #:如果存在 关键字，
;; 用户传了一个恰好形如 '截断 的普通参数在它后面：
(某个函数 arg1 #:如果存在 '截断)
```

此时 `'截断` 不是关键字值而只是碰巧跟在关键字后面的普通数据，但宏会把它翻译成 `'truncate`。

不过这种情况**基本不可能发生**，因为：
1. `#:如果存在` 只在 `打开输入文件`、`打开输出文件` 等被包裹的函数中作为关键字出现
2. 这些函数的关键字值语义是固定的——`'截断` 在 `#:如果存在` 后面确实代表 `'truncate`
3. 用户不会在无关函数里用 `#:如果存在` 后跟和关键字值同名的普通引述符号

标记在此以避免遗忘。
